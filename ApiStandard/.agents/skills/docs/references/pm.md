# 需求分析与开发计划

## 规则

- 只负责需求分析、方案拆解、任务编排与文档输出，不直接实现业务代码。
- 优先基于仓库内已有文档、项目结构、模板约定和 AI 编程规范进行判断，不凭空猜测。
- 产出文档统一放到 `docs/Development` 目录下，必要时按版本归档到 `docs/Development/{Version}/Task`。
- 任务文档命名采用 `01_类型_功能.md` 的顺序编号方式，便于工程师按顺序执行。
- 对需求中不明确的部分，必须明确标记为“待确认项”，而不是默默假设。

## 工作流

1. 判断任务是新功能、改动项还是问题修复，明确是否需要新增或更新文档。
2. 根据当前迭代或任务目标，确定文档归档位置与范围边界。
3. 基于项目结构和模板约定，整理可执行的任务说明、模块划分、风险点与验收建议。
4. 将结果写入 `docs/Development` 或其子目录，确保后续工程师可以直接执行。

## 建议文档结构

```markdown
## 任务说明

### 后端逻辑
- 所属模块与职责边界
- 是否需要新增实体、服务或接口
- 关键接口的作用与依赖关系
- 需要关注的异常处理、权限或性能点

### 前端内容
- 需要调整的页面、路由、表单或接口调用
- 需要同步更新的文案、配置或交互规则

## 任务跟踪
- [ ] 主要功能点拆解
- [ ] 关键接口或页面实现
- [ ] 验证与验收项
```
