# 通用 UX 规范

## 1. 信息层级与布局

- 先呈现页面目的，再呈现关键内容和主要动作；用标题、分组、间距和对齐表达层级，不依赖大量边框和装饰卡片。
- 使用一致的间距尺度和设计 token。不要在页面内散落任意像素、颜色、圆角和阴影。
- 控制内容行宽，避免宽窗口中表单控件、按钮和正文无意义地拉满。表单按阅读顺序组织，相关字段靠近，不相关字段分节。
- 保留稳定的动作位置。数据加载、校验错误或辅助文本出现时，避免让主要按钮突然移动。
- 为长文本、200% 文本缩放、RTL/本地化和高 DPI 留出空间；不要以截断代替必要内容。

## 2. 控件选择

| 用户意图 | 首选模式 | 避免 |
|---|---|---|
| 执行一次动作 | Button / Command | 用 Switch、Checkbox 或可点击文本冒充按钮 |
| 在多个对象中选择任意项 | Checkbox | 用多个独立 Switch |
| 从一组互斥项中选择一个 | Radio；选项很多时用 Select/ComboBox | 用多个 Checkbox |
| 立即改变一个持续设置 | Switch/ToggleSwitch | 需要“保存”后才生效却使用 Switch |
| 页面或资源间导航 | Link、Navigation item | 用普通 Button 模拟链接且破坏返回行为 |
| 录入已知格式值 | 对应的日期、时间、数字或选择控件 | 让用户在纯文本框中猜格式 |
| 展示可比较的结构化数据 | Table/DataGrid | 用卡片堆叠大量同构字段 |
| 展示对象摘要与少量动作 | Card/List item | 把整个复杂表单放进卡片 |
| 需要阻断流程的重要决定 | Dialog | 用弹窗展示普通说明或可内联编辑内容 |
| 短暂、非阻断反馈 | Snackbar/Toast/状态区 | 用 toast 承载必须阅读或唯一的错误信息 |

控件标签描述结果或对象，例如“保存更改”“删除订单”，不要使用含糊的“确定”“点击这里”。图标按钮必须有可访问名称；陌生图标同时提供文本或 tooltip。

## 3. 状态与反馈

- 加载：首次加载使用与内容结构匹配的占位或明确进度；局部刷新保留已有内容；无法估计时不要伪造百分比。
- 空状态：区分“从未有数据”“筛选无结果”“无权限”和“加载失败”，分别提供创建、清除筛选、申请权限或重试动作。
- 提交：开始后禁用重复提交并显示进行中状态；完成后明确成功结果；失败时保留用户输入和上下文。
- 校验：在相关字段附近展示具体、可修复的错误，同时在提交后把焦点或摘要引导到第一个错误。不要只变红，也不要在用户尚未完成输入时过早报错。
- 删除与不可逆操作：明确对象和影响，危险按钮使用清晰文案；能可靠撤销时优先“执行后撤销”，真正不可逆或高影响操作才要求确认。
- 禁用：只有当前确实不可用时禁用，并让原因可发现；若用户需要理解功能，考虑保持可操作并在尝试时解释前置条件。

## 4. 可访问性基线

- 保证逻辑阅读顺序、可见焦点、键盘或辅助技术可到达全部功能；不创建键盘陷阱。
- 正文文字与背景至少 4.5:1，大号文字至少 3:1；组件边界和状态也需有足够非文本对比。焦点样式不能仅是轻微颜色变化。
- Web 指针目标至少满足 WCAG 2.2 的 24×24 CSS px 或间距例外；触摸平台按自身更严格规范执行。不要把 24px 当成推荐的手机触控尺寸。
- 使用文本加形状/图标表达错误、成功、选中和警告，不能只依赖红绿颜色。
- 尊重系统的字体缩放、减少动画、高对比度、主题和输入设备设置。动画应解释空间关系或状态变化，不能延迟任务；提供 reduced-motion 行为。
- 为输入、图片、图表、状态消息和自定义控件提供名称、角色、值和必要描述。装饰图形从辅助树中移除。

## 5. 交互一致性

- 返回保持用户心智模型：关闭临时层、回到上一层、返回列表时尽量保留滚动、筛选和选择状态。
- 不隐藏关键动作于仅 hover、右键、长按或手势中；这些只能作为快捷方式，并提供可发现的等价入口。
- 避免自动提交、自动导航和意外焦点跳转。输入造成重大变化前让结果可预期。
- 使用熟悉的平台术语和顺序；同一产品中相同动作保持相同标签、图标、位置和快捷键。

## 6. 审查清单

- 用户能否在数秒内辨认页面目的和主要动作？
- 每个控件是否表达正确的选择或动作语义？
- 加载、空、无结果、错误、权限不足和提交中的表现是否明确？
- 是否能只用键盘或目标设备的主要输入方式完成任务？
- 200% 文本缩放、长翻译和窄/宽窗口下是否仍可用？
- 错误能否被发现、理解并修复？危险操作是否可恢复或被适当确认？
